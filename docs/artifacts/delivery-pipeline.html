GATE 4 Open the PR?
you decide orchestrator opens it + sets board board-keeper · Sonnet 5

After push is confirmed. Once approved, the PR is opened and the board moves to Code Review immediately after — then board-keeper is dispatched to re-query that write and confirm it actually stuck (gh project item-edit has silently failed before), sweeping the rest of the board for unrelated drift while it's there: flags any open issue/PR missing a label, and safely promotes to Done any closed parent issue whose sub-issues all trace to their own merged PRs.

A third trigger, off this ticket's rail — board-keeper also fires the instant any new issue is filed (gh issue create), wherever that happens mid-flow. It confirms the issue landed on the board with a status and a label instead of waiting for the next full sweep to notice.

GATE 5a Watch CI to completion
ci-watcher · Haiku 4.5 orchestrator diagnoses on failure

Dispatched in the background with the PR's number, so the polling noise lands in its own disposable context instead of the orchestrator's — and the session isn't blocked while checks run.

If a check fails — required or not — it hands back the check name, run URL, and log excerpt. Orchestrator diagnoses, delegates the fix to the owning layer subagent, then Gate 3 fires again for the fix-push.
