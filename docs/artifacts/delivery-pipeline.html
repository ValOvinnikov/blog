From picking a ticket to a merged PR

Nine stages, three actors doing the work — the orchestrating session, scoped subagents on their own models, and plain mechanical commands — plus four moments that stop and wait for you. Everything else runs without asking.

The full roster, by scope

Every subagent that can appear in a run, grouped by what it's responsible for — not every stage above uses all of them.

layer
layer subagent(s) — config / cms / service / ui / web · Sonnet 5

packages/config · apps/cms · packages/service · packages/ui · apps/web

Each owns exactly one workspace, in its own isolated worktree. Dependency order runs config → cms → service → ui → web — only the layers a change actually touches get dispatched.

review
reviewer · Sonnet 5

full diff, every issue

Fresh eyes on the whole change — mechanical scan, layer-contract pass, general review. Gate 1 doesn't complete without an APPROVE from this one.

a11y-reviewer · Sonnet 5

packages/ui · apps/web only

Accessibility-specific pass — ariaLabel conventions, real heading tags, focus-visible, alt text. Only dispatched when the diff touches UI or web presentation.

seo-auditor · Sonnet 5

apps/web only

Metadata, structured data, sitemap/robots/RSS — dispatched only when a diff touches routes, metadata, or feeds.

quality
test-writer · Sonnet 5

*.test.ts(x) only, every layer

Adds or extends test coverage after the layer agents finish. Scope-guarded — a fix it needs to make outside a test file comes back as a finding, never an edit it makes itself.

verify-runner · Haiku 4.5

Gate 1, before reviewer runs

Runs the integration pass — type-check, lint, test, build — and reports pass/fail only. Dispatched synchronously (Gate 1 can't proceed without the result), so the bulky turbo output lands in its own context, not the orchestrator's.

ops
ci-watcher · Haiku 4.5

Gate 5a, after the PR opens

Polls the PR's checks to a terminal state in the background and reports pass/fail — same cost-isolation reasoning as verify-runner, just later in the pipeline and non-blocking since other work can continue while it waits.

board-keeper · Sonnet 5

the whole board, not just the current issue

Reconciles board status against repo reality after every PR open/merge — re-queries its own writes to confirm they stuck, and surfaces anything destructive-looking (like a wrongly-closed issue) instead of touching it.

discovery
explore · Haiku 4.5

on demand, before planning

A cheap, disposable sweep for "where does X live" / "is there already a Y" questions — burns a Haiku context so that rediscovery never has to enter the orchestrator's own.
